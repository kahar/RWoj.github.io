Preferences

For a better description of what kind of programmer I am, I will describe my dream project:

  • backend and java take at least 70% of the working time
  • Spring framework - I don't mind other tools, I'm just the most familiar with this tool
  • modularity
  • unit/integration tests that are written on many levels - not only classes, but also entire modules, or at the controller level - I wrote unit / integration tests because there are many approaches to them and the definition boundaries are fluid
  • end to end tests
  • significant coverage of business logic tests - if I make a bug and all tests at all levels are passed, and it will be deployed to production - then it is unlikely that I will get a call, that something should be corrected ASAP
  • on frontend side there is something like MUI or PrimeFaces - because that allow me and team to create a consistent and aesthetic GUI faster and easier
  • impact on the project - when tasks are described precisely at the business level, but leave space at the technical level
  • a team of people who have more or different experience than me
  • includes a test and pre-production environment - on which individual branches or features can be tested

A list of things I don't like that I have encountered:

  • test with data import to db from csv - generally this is not a bad idea, but the problem starts when it's the default way to create tests
  • slow tests - if there are no quick tests that give answer that everything is rather fine after the changes
  • Waterfall Project - it has its right uses, but these are not projects that I would like to participate in
  • regular over hours - sometimes overtime may be required. However, I do not like when they are predictable or regular, for example: deployments in pre-production test environments in the late afternoon - which someone has to supervise
  • custom solutions - for something which can be handled by libraries when this custom solution has no advantage over the library. For example: the implementation of batch processes
  • it is not possible to deploy my own branch to some test environment - I can only test on a local machine, then the branch is merged to the main branch - this forces corrections to be made ASAP
  • fight with machines - when I have to solve infrastructure problems more than once every three months (locally or in other environments) to be able to solve business problems
